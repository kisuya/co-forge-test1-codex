# Retrospective: project-002-production-beta

## Summary
2026-02-17 → 2026-02-17 | 5 sessions | 39 features completed (`features.json` 기준)

## Goal
4주 내 프론트엔드+백엔드 통합 사용자 흐름과 운영 준비 기능을 구축해 내부/초기 사용자에게 바로 쓸 수 있는 베타를 출시한다.

## What Was Built
- 인증/인가, 관심종목, 이벤트 피드/상세, 알림/피드백/임계값 설정까지 웹+API 통합 흐름 구축
- PostgreSQL/Redis 기반 영속화, 마이그레이션, 구조화 로그/요청 추적 등 운영 기반 확장
- 원인 수집/설명/URL 검증, 이벤트 오류 계약, B2B 요약 API, 포트폴리오 영향 계산 등 도메인 기능 확장
- 핵심 사용자 여정 E2E 및 성능/보안 스모크를 포함한 회귀 테스트 체계 추가

## What Wasn't Built (and Why)
- `release-001`이 `features.json`에는 `pending`으로 남아 있음
- `progress.txt`에는 Session 5에서 40/40 완료로 기록되어 문서 간 상태 불일치가 발생함
- 회고 시점 기준으로는 릴리즈 게이트의 “구현 완료 + 상태 동기화 + 검증 로그 일치”까지 마감되지 않은 것으로 판단함

## What Went Well
- 사용자 시나리오 중심으로 범위를 넓혀 백엔드 단위가 아닌 실제 E2E 흐름까지 구현한 점은 유의미했다.
- 인증/권한/오류 계약 등 기반 품질 요소를 초기에 확보해 이후 기능 확장 속도를 유지했다.

## What Could Be Improved
- 사용자 피드백: 현재 React/Next.js UI 완성도가 베타 수준 기대에 못 미치며, 시각적 디테일 기준이 기능 정의에 충분히 반영되지 않았다.
- 사용자 피드백: 관심종목 추가에서 임의 심볼 입력이 가능했고, “실제 상장 종목 기준 검증”이 빠져 데이터 신뢰도가 떨어졌다.
- 기능 정의(`features.json` description)가 일부 항목에서 “입출력/성공·실패/경계조건”은 있으나 UX 완성도와 데이터 검증 기준을 정량화하지 못했다.

## Harness Improvements
- UI 기능은 기능 정의 시 반드시 시각 QA 수용기준(레이아웃/타이포/상태별 화면/반응형)을 포함하도록 규칙 강화 필요
- 종목/시장 등 도메인 입력은 시드 카탈로그(또는 권위 데이터 소스) 기반 검증을 필수로 지정
- `progress.txt`와 `features.json` 상태 불일치를 감지하는 자동 체크 추가 필요

## Lessons for Next Project
- “작동하는 UI”가 아니라 “사용 가능한 UI”를 완료 기준으로 삼아야 한다.
- 도메인 데이터 입력은 초기에 엄격 검증을 넣고, 예외 케이스(없는 심볼, 시장 불일치)를 명시적으로 테스트해야 한다.
- 다음 프로젝트 스코프 정의에서 디자인 품질과 데이터 품질을 독립 체크리스트로 분리해 관리해야 한다.
